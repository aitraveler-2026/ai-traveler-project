# 구현 범위

여기어때 검색플랫폼 개발팀 **Search Platform Engineer** 채용 과제의 구현 범위를 정의합니다.
*무엇을 만들어야 하는가* 에 집중한 문서이며, 진행 방법·평가 시각·일정 등 운영 안내는 함께
제공되는 [`가이드.md`](가이드.md) 를, 데이터 필드는 [`data-spec.md`](data-spec.md) 를 참고하세요.

---

## 1. 배경

여기어때는 모텔·호텔·펜션 등 숙박 시설을 예약하는 OTA(Online Travel Agency) 입니다.
사용자가 검색을 시작하는 첫 순간이 **검색창**이고, 글자를 입력하는 즉시 떠오르는
**자동완성(autocomplete)** 추천이 검색 경험의 품질과 전환율을 좌우합니다. 한 글자마다
수십 ms 안에, 의미 있는 후보를, 적절한 순서로 보여줘야 합니다.

검색플랫폼팀은 이 자동완성을 *서비스로서* 만들고 운영합니다. 단순히 "되는" 것을 넘어
**빠르고(latency), 안정적이고(stability), 품질 좋은(quality)** 자동완성을 어떤 자료구조·
저장소·검색엔진 위에 어떻게 설계할지가 이 직무의 핵심 질문입니다.

이 과제에서는 *행동 로그가 없는 환경*에서, **숙소 카탈로그만으로 자동완성 서비스를
어떻게 설계·구현할지** 를 묻습니다.

## 2. 과제

### 2.1 메인 (필수)

**3,000개 한국 숙소 데이터를 바탕으로, 사용자가 검색창에 입력하면 실시간으로 후보를
추천하는 자동완성 서비스를 설계·구현하세요. 백엔드 API 와 프론트엔드(검색창 + 추천
드롭다운) 가 모두 동작해야 합니다.**

최소 동작 정의(필수):
- 사용자가 **숙소명(name)** 의 앞부분을 입력하면, 그 prefix 로 시작하는 숙소명을 추천한다.
- **백엔드**: 입력 문자열을 받아 추천 후보 목록을 반환하는 검색 API (Java/Spring).
- **프론트엔드**: 검색창에 타이핑하면 추천 목록이 실시간으로 갱신되는 UI.

핵심 질문:
- 추천 후보를 **어떤 자료구조·저장소**(인메모리 trie / Redis / 검색엔진의 completion·edge-ngram 등)
  위에 올릴 것인가, 그 선택의 *근거*는 무엇인가
- 같은 prefix 에 여러 후보가 걸릴 때 **어떤 순서**로 보여줄 것인가 (랭킹 기준)
- **응답 지연·동시성·데이터 갱신**을 어떻게 다룰 것인가
- 어떤 한계가 있고, 다음에 무엇을 개선하겠는가

### 2.2 가점 영역 (선택)

아래는 *모두 선택*입니다. 무엇을 골라 얼마나 깊이 들어갈지가 본인의 판단이자 평가 신호입니다.

- **추천 소스 확장** — 숙소명뿐 아니라 지역(시도·시군구·동·랜드마크), 키워드(시설·분위기 태그)
  까지 추천. 서로 다른 타입의 후보를 한 목록에서 *어떻게 구분·정렬*할지가 관전 포인트.
- **한글 고급 입력 처리** — 초성 검색(`ㄱㅈㅁ` → `강제매뉴`), 자모 단위 prefix(「강」 입력 도중 「ㄱ」/「가」),
  영한 오타 교정(`rkdska` → `강남`), 오타 허용(편집거리). *한국어 검색 플랫폼의 핵심 변별 영역*입니다.
- **추천 랭킹 고도화** — prefix match 점수 + 인기도 신호. 행동 로그가 없으므로 인기도를
  `review_count`·`rating` 같은 proxy 로 설계하거나, 본인의 가정 하에 검색·클릭 로그를 임의 생성해 활용.
- **인덱싱·갱신 파이프라인** — 카탈로그가 바뀔 때 추천 인덱스를 어떻게 (재)구축·갱신할지.
- **성능·운영 관점** — 응답 지연 측정, 부하·동시성 고려, 캐싱, 관측(로깅·지표) 등.

> 가점은 *기능을 많이 붙였는지*가 아니라, **선택한 항목의 설계 근거·트레이드오프·결과
> 해석**을 봅니다. 하나를 깊게 파는 것이 여럿을 얕게 붙이는 것보다 좋은 신호입니다.

## 3. 제공 데이터

### 3.1 숙소 카탈로그 — `lodgings.jsonl` (3,000줄)

한국 숙소 3,000개의 콘텐츠 정보. 카테고리(모텔·호텔·펜션), 위치(시도·시군구·동·좌표·인근 랜드마크),
가격, 평점·리뷰 수, 시설 multi-hot, 분위기 태그, 텍스트 3종, 오픈 연도·최근 리뷰일이 포함됩니다.

자동완성 관점에서 자주 쓰일 필드:
- `name` — **추천의 최소 필수 소스**
- `location.sido` / `sigungu` / `dong` / `nearest_landmark` — 지역 소스 확장(가점)
- `facilities` / `vibe_tags` — 키워드 소스 확장(가점)
- `rating.score` / `rating.review_count` — 인기도 proxy 후보(가점 랭킹)

> **사용자 행동 로그(검색·클릭·예약)는 제공되지 않습니다.** 추천 순서에 인기도를 쓰고
> 싶다면 카탈로그 내 신호를 proxy 로 쓰거나, 본인의 가정 하에 직접 생성하세요(가점).

상세 필드 스펙은 함께 제공되는 [`data-spec.md`](data-spec.md) 를 참조하세요.

## 4. 제출물

### 4.1 코드

누가 와도 재현 가능한 형태로 정리해주세요:
- `README.md` — 환경 설정·빌드·실행 방법 (백엔드 + 프론트엔드 모두)
- 의존성 명세 (`build.gradle` / `pom.xml` 등)
- 실행 가능한 백엔드(Java/Spring) + 프론트엔드. 로컬에서 검색창에 타이핑하면 추천이 떠야 합니다.

> 언어는 **Java/Spring** 으로 고정합니다(팀 실무 스택). 검색엔진·저장소·프론트엔드
> 프레임워크는 자유롭게 선택하세요 — *무엇을 왜 골랐는지*가 평가의 일부입니다.

### 4.2 리포트 (`report.md` 또는 `report.pdf`)

자유 형식입니다. 순서·섹션 분할은 본인 판단으로. 다음 내용을 담아주세요:

- 문제를 어떻게 정의했고 어떤 가정을 했는가
- 어떤 자료구조·저장소·검색엔진·아키텍처를 선택했고 그 *근거*는 무엇인가 (대안과의 비교 포함)
- 응답 지연·메모리·확장성·정확도 사이의 *트레이드오프*를 어떻게 인식·결정했는가
- 결과의 *한계*와 다음에 개선한다면 무엇을 해보겠는가
- AI 도구(Claude Pro)를 어떻게 활용했는가 — 어떤 도움을 받았고 어디서 비판적으로 검증했는가

### 4.3 AI 활용 섹션 (필수)

리포트 안에 별도 섹션으로 다음을 포함해주세요:

- 어떤 작업·결정에 AI 도구를 사용했는지 (목록)
- AI 출력을 *그대로 채택 / 수정 / 거부* 한 사례 각각 최소 1건 (근거 포함)
- AI 도구를 쓰면서 발견한 한계·실수 최소 1건

본인이 가장 잘 활용했다고 생각하는 Claude 대화 transcript 1~2건을 함께 제출해도 좋습니다 (필수 아님).

## 5. 범위 안 / 범위 밖

명시적으로 변별을 위해 적습니다. 애매하면 본인 판단을 *리포트에 적고* 진행하세요 —
판단 근거가 좋으면 그 자체가 평가 신호입니다.

**범위 안**
- 자동완성 백엔드 API(Java/Spring) + 프론트엔드(검색창·추천 드롭다운)의 *설계·구현*
- 추천 자료구조·저장소·검색엔진 선택 (인메모리·Redis·Elasticsearch 등 자유)
- 추천 후보 랭킹(정렬 기준) 설계
- 데이터 전처리·인덱스 구축·결측 처리
- 가점: 추천 소스 확장 / 한글 고급 입력 처리 / 랭킹 고도화 / 인덱싱·갱신 / 성능·운영
- 외부 공개 라이브러리·검색엔진 활용 (출처·이유 명시)

**범위 밖**
- 응답 지연·정확도의 *절대 점수 경쟁* (정량 비교 채점 없음 — 정성 평가만)
- 실서비스 배포·CI/CD·인프라 프로비저닝 구축
- 인증·결제·예약 등 자동완성 외 기능
- 운영 환경 모니터링 대시보드·A/B 테스트 *구축* (성능·관측을 *설계 관점*에서 논하는 것은 범위 안)
- 우리(여기어때) 내부 데이터·시스템에 대한 접근 가정
